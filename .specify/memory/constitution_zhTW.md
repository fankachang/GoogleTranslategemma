# TranslateGemma 專案憲法

## 核心原則

### 一、本地化優先（不可協商）
所有規格文件、計畫文件與使用者文件**必須**以繁體中文（zh-TW）撰寫。Git commit 訊息、程式碼註解與內嵌文件**必須**使用繁體中文。唯有憲法本身以英文撰寫，並且**必須**在同一目錄下提供 `constitution_zhTW.md` 繁體中文版本。

**理由**：本專案主要服務繁體中文使用者。一致的本地化確保可及性、可維護性與使用者基礎的一致性。僅有英文的技術文件會為開發團隊與專案關係人帶來障礙。

### 二、API 契約完整性
當前端呼叫後端 API 或函式時，該 API/函式**必須**在後端實際存在。前端開發人員**不得**任意命名、假設或虛構 API。所有 API 定義在整合前**必須**經過後端實作驗證。

**理由**：假設性的 API 會導致執行期錯誤、整合延遲與脆弱契約。明確驗證強化紀律並降低微服務架構（Blazor WASM ↔ FastAPI）中的除錯成本。

### 三、簡潔與 YAGNI（不可協商）
避免過度設計與過度工程。功能**必須**解決經驗證的即時問題。投機性抽象、過早最佳化與不必要的複雜性皆被禁止，除非有明確理由與核准。

**理由**：專案明確聲明「設計原則：簡單為主」。這是一個專注的翻譯服務，沒有使用者帳號或持久化機制。功能蔓延會破壞可維護性與部署速度。

### 四、測試先行開發
所有新功能與錯誤修復在實作前**必須**包含自動化測試。測試**必須**涵蓋：
- 業務邏輯的單元測試（後端模型推論、前端服務）
- API 契約的整合測試（後端路由 ↔ 前端服務）
- 關鍵使用者流程的端對端測試（翻譯提交、串流回應、語言選擇）

測試**必須**在實作開始前撰寫、審查、核准並處於失敗狀態。遵循紅-綠-重構循環。

**理由**：專案已建立 `backend/tests/unit/` 與 `backend/tests/integration/` 目錄。測試先行開發可防止迴歸、記錄意圖，並在 GPU 加速 ML 推論管線中安全重構（除錯成本高昂）。

### 五、UI 版面一致性
前端 UI 元件**必須**考慮一致的版面高度。輸入欄位、聊天泡泡及其佈局**必須**保持視覺協調。元件高度、內距與間距**必須**遵循統一設計系統（程式碼庫中記錄的 MudBlazor 主題客製化）。

**理由**：開發指南明確規定。不一致的 UI 會降低使用者體驗，並在擴展元件庫時產生技術債。

### 六、虛擬環境衛生
在創建 Python 虛擬環境（如 `.venv`）前，**必須**檢查專案根目錄是否已存在。若存在則重複使用。**不得**創建重複或巢狀虛擬環境。

**理由**：開發指南明確規定。重複環境會造成依賴衝突、浪費磁碟空間，並使 CI/CD 管線複雜化。

### 七、可觀測性與可除錯性
所有後端 API 端點**必須**實作結構化日誌（JSON 格式）並包含請求 ID 追蹤。模型推論**必須**記錄：
- 使用的裝置（CUDA/MPS/CPU）
- 詞元數量（輸入/輸出）
- 延遲指標（詞元化、推論、後處理）

前端**必須**將 API 呼叫失敗完整錯誤上下文（狀態碼、回應主體、請求載荷）記錄至瀏覽器主控台。

**理由**：TranslateGemma 推論涉及 GPU 加速、串流回應與多語言詞元化。沒有結構化日誌無法除錯生產問題。SSE 串流失敗在沒有客戶端日誌時極難診斷。

## 技術限制

### 核准技術棧
- **前端**：Blazor WebAssembly (.NET 9)、MudBlazor UI 函式庫
- **後端**：Python 3.13（PyTorch CUDA wheels 必需）、FastAPI、Transformers（Hugging Face）
- **模型**：TranslateGemma 4B / 12B（透過 `config.yaml` 設定）
- **容器化**：Podman / Docker，多階段建置
- **GPU 支援**：NVIDIA CUDA 12.4+、Apple MPS、CPU 後備

核心技術棧變更需經架構審查與遷移計畫核准。

### 禁止模式
- 禁止客戶端狀態持久化（localStorage/sessionStorage），僅限當前頁面 session
- 禁止使用者驗證或帳號管理
- 禁止資料庫或持久化儲存層
- 禁止客戶端模型推論（WASM 限制禁止 TranslateGemma）

## 開發工作流程

### 程式碼審查關卡
所有 Pull Request **必須**通過：
1. 自動化測試套件（後端 `pytest backend/tests/`、前端 `dotnet test`）
2. Linting（Python: `ruff`、C#: 內建分析器）
3. 憲法合規檢查（審查者手動驗證）
4. 至少一位維護者的核准審查

### 容器化標準
所有 Containerfile **必須**：
- 使用多階段建置以最小化映像大小
- 固定基礎映像版本（如 `mcr.microsoft.com/dotnet/aspnet:9.0.1`）
- 宣告 HEALTHCHECK 端點
- 在註解中記錄 GPU 直通需求
- 遵循 `.github/skills/container-image-hygiene/SKILL.md` 指引

### 設定管理
執行期設定**必須**外部化至 `config.yaml`。禁止硬編碼路徑、模型名稱或裝置選擇。Schema **必須**在啟動時驗證，並大聲拒絕無效設定（快速失敗）。

## 治理

本憲法優於所有衝突的實踐、指南或偏好。修訂需要：
1. 記錄理由與架構影響分析
2. 專案維護者核准
3. 受影響程式碼/文件的遷移計畫（若為破壞性變更）

所有 PR 與程式碼審查**必須**驗證憲法合規性。複雜性**必須**依據原則三（簡潔與 YAGNI）提出理由。

執行期開發指引與操作程序請參考 `AGENTS.md` 與 `.github/copilot-instructions.md`。
