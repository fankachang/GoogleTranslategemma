# Feature Specification: 可設定的翻譯輸入字數上限

**Feature Branch**: `001-input-max-length`  
**Created**: 2026-02-19  
**Status**: Draft  
**Input**: User description: "前端頁面的翻譯輸入欄位需要追加輸入字數上限功能，此限制需要可以在 config.yaml 中進行設定，預設值為 512 字元。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 輸入框即時字數提示與限制 (Priority: P1)

使用者在翻譯輸入框中輸入文字時，頁面會即時顯示目前字數與上限，並在超過上限時以醒目顏色警告、停用送出按鈕，防止使用者送出無效請求。

**Why this priority**: 這是使用者直接感受到的體驗核心。沒有即時提示，使用者不知道自己快要超出限制；沒有停用送出，後端才報錯會造成多餘的往返延遲。此故事可獨立交付並為使用者提供直接價值。

**Independent Test**: 開啟網頁，在輸入框輸入任意文字，觀察右下角字數計數器是否隨輸入更新，並驗證輸入達上限時計數器是否變紅、送出按鈕是否停用。

**Acceptance Scenarios**:

1. **Given** 使用者開啟翻譯頁面，**When** 輸入任意文字，**Then** 輸入框下方應即時顯示「目前字數 / 上限字數」（例如 `45 / 512`）
2. **Given** 輸入文字尚未達上限，**When** 字數在正常範圍內，**Then** 字數顯示應使用預設色（不警告）
3. **Given** 輸入文字長度超過上限，**When** 字數超過上限值，**Then** 字數顯示應變為警告色（紅色），且送出按鈕應被停用
4. **Given** 送出按鈕已因超限而停用，**When** 使用者刪除文字使字數回到上限以內，**Then** 送出按鈕應恢復可用狀態
5. **Given** 使用者輸入框為空，**When** 頁面載入完成，**Then** 字數顯示應為 `0 / 512`（或設定的上限值）

---

### User Story 2 - 管理員透過 config.yaml 設定字數上限 (Priority: P2)

服務管理員在部署或日常維運時，可以在 `config.yaml` 中設定翻譯輸入的最大字元數，重啟服務後新限制即生效，前後端皆套用同一設定值。

**Why this priority**: 這是本功能的核心可設定性需求。預設值 512 可直接滿足一般使用情境，但不同部署場景（如企業翻譯用途）可能需要更高上限，因此可設定性是必要的。P2 排序是因為若只實作 P1（固定 512），系統仍具備最低可用性。

**Independent Test**: 修改 `config.yaml` 中的字數上限為 256，重啟服務後開啟網頁，確認輸入框顯示的上限為 256，且輸入 257 個字元時送出按鈕被停用。

**Acceptance Scenarios**:

1. **Given** `config.yaml` 中未設定字數上限，**When** 服務啟動，**Then** 系統應使用預設值 512 字元作為輸入上限
2. **Given** 管理員將 `config.yaml` 中的字數上限設為 256，**When** 重啟服務後開啟網頁，**Then** 輸入框的字數上限應顯示為 256
3. **Given** 管理員設定字數上限為 1024，**When** 使用者輸入 800 字元，**Then** 字數顯示不應觸發警告，且送出按鈕保持可用
4. **Given** 管理員設定字數上限為 1024，**When** 使用者輸入 1025 字元，**Then** 字數顯示應觸發警告，且送出按鈕應被停用

---

### User Story 3 - 後端拒絕超出上限的請求 (Priority: P3)

即使前端未正確攔截（例如 API 被直接呼叫），後端也會驗證請求內容的字元數，超出上限時回傳清楚的錯誤回應，並在前端介面上顯示友善的錯誤訊息。

**Why this priority**: 前端防護可能被繞過（如直接呼叫 API），因此後端的獨立驗證是必要的防線，確保資源不被濫用。此故事獨立可測，且可單獨驗證後端行為。

**Independent Test**: 透過 API 工具直接向後端送出超過設定上限字數的翻譯請求，確認後端回傳 4xx 錯誤狀態碼與可讀的錯誤訊息，且前端在此情況下能顯示錯誤提示而非靜默失敗。

**Acceptance Scenarios**:

1. **Given** 後端設定上限為 512 字元，**When** 收到含有 513 字元輸入文字的翻譯請求，**Then** 後端應拒絕請求並回傳 HTTP 422 狀態碼，回應 body 格式為 `{"detail": "輸入文字超過允許上限（512 字元）"}`
2. **Given** 後端拒絕超出上限的請求，**When** 前端收到此錯誤回應，**Then** 前端應在對話介面上顯示友善的錯誤訊息泡泡（如「輸入文字超出字數限制」）
3. **Given** 後端設定上限為 512 字元，**When** 收到恰好 512 字元的翻譯請求，**Then** 後端應正常處理並回傳翻譯結果（邊界值應被接受）

---

### Edge Cases

- 使用者貼上大量文字（如複製整份文件），字數瞬間超過上限時，前端字數顯示與送出按鈕狀態應立即反映？
- 管理員將字數上限設為 0 或負數等無效值時，系統應回退至預設值 512，而非讓服務無法使用
- 不同語言的字元（如中文、日文、阿拉伯文等多位元組字元）的計數方式：以 Unicode 字元（程式碼點）為單位，不以位元組為單位

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一個可在 `config.yaml` 的 `translation` 區段設定翻譯輸入上限的欄位，預設值為 512 字元
- **FR-002**: 前端輸入元件 MUST 在使用者輸入時即時顯示「目前字數 / 上限字數」
- **FR-003**: 前端 MUST 在輸入字數**超過**上限時，將字數顯示切換為警告狀態（視覺區分）；恰好等於上限時不觸發警告色（無近上限預警機制）
- **FR-004**: 前端 MUST 在輸入字數超過上限時停用送出按鈕，並在字數回到上限以內時自動恢復
- **FR-005**: 後端 MUST 對每個翻譯請求驗證輸入文字長度，拒絕超過設定上限的請求，回傳 HTTP **422 Unprocessable Entity**，回應 body 格式為 `{"detail": "輸入文字超過允許上限（{limit} 字元）"}`（`{limit}` 替換為實際設定上限值）
- **FR-006**: 前後端所使用的字數上限值 MUST 來自同一設定來源（`config.yaml`），以確保兩側限制一致
- **FR-007**: 當前端收到後端的超限錯誤回應時，MUST 在對話介面上顯示使用者可讀的錯誤訊息
- **FR-008**: `config.yaml` 中若未設定上限欄位，系統 MUST 使用預設值 512 字元，服務啟動不得失敗
- **FR-009**: 字元計數 MUST 以 Unicode 字元（程式碼點）為單位，不得以位元組為單位
- **FR-010**: `/api/config` 端點 MUST 為公開可讀（不需驗證），任何用戶端無需 token 或 session 即可 GET 取得設定值
- **FR-011**: 前端初始化時若 `/api/config` 呼叫失敗（網路錯誤或非 2xx 回應），MUST 自動回退使用預設值 512，不阻擋使用者操作；錯誤可靜默記錄至 console，無需顯示 UI 錯誤訊息

### Key Entities

- **翻譯輸入上限設定**：`config.yaml` 中 `translation` 區段的一個整數欄位，代表允許的最大輸入字元數；預設值 512，有效範圍為正整數
- **字數計數顯示**：前端輸入元件中顯示「目前字數 / 上限字數」的 UI 元素，隨輸入即時更新，超限時切換至警告樣式

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者在輸入任何文字時，字數計數器在 100ms 以內更新（即時反映）
- **SC-002**: 輸入字數達到上限後，送出按鈕停用狀態在 100ms 以內生效，無需使用者額外操作
- **SC-003**: 管理員修改 `config.yaml` 並重啟服務後，前後端皆在下一次頁面載入時套用新的上限值，無需修改任何程式碼
- **SC-004**: 後端對超出上限的請求回傳明確的錯誤回應，前端能正確顯示友善錯誤訊息，使用者不會看到空白或系統級的技術錯誤
- **SC-005**: 字數上限設定為 1 到 10,000 之間的任意整數時，前後端均能正確套用，不會發生服務異常

## Assumptions

- 字元計數採用 Unicode 程式碼點（code points），與大多數文字編輯器的計數方式一致
- 前端取得上限值的方式：後端新增輕量端點（`/api/config`），前端於應用程式初始化時以 HTTP GET 呼叫取得；此端點**無需驗證**（公開可讀），確保 FR-006 的單一設定來源要求，不需在部署時手動同步前後端設定
- 若 `/api/config` 呼叫失敗，前端靜默回退至預設值 512，不阻擋任何使用者操作
- `config.yaml` 中設定值若為 0 或無效（非正整數），系統回退至預設值 512，服務繼續正常運作
- 使用者介面的警告樣式（顏色等）沿用現有 MudBlazor 主題的 Error 色系，與目前字數顯示的既有實作保持一致
- 不設近上限預警機制（如黃色提示色），僅在字數**超過**上限時才切換警告色
